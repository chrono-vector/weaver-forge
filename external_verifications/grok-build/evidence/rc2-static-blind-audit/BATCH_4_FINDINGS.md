# RC2 static blind audit — Batch 4 findings: Validator, classification, policy

Batch 4 of 4. Static review only; no execution performed. Scope: template/generator/validator consistency, manifest and verdict parsing strictness, classification precedence, and package/process completeness for tag `grok-build-witness-v1.0.0-rc2` (`255b357c9ee33c4a9e34b5d9b6e396c53cfe494e`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RB-019 | Templates vs generators vs validator field-name drift | Field names used in evidence templates, in the scripts that generate evidence, and in the validator that checks evidence are not kept in a single source of truth, and have drifted relative to one another. | A submission can pass generation and still fail (or wrongly pass) validation purely due to naming inconsistency rather than substantive defects. |
| RB-020 | No file-specific schema validation | The validator checks that required files are present but does not enforce a schema specific to each file's expected fields, so a generic body could satisfy a check intended for an unrelated file. | The validator can be satisfied by evidence that does not actually contain the content it is supposed to attest to. |
| RB-021 | Manifest grammar too loose | The evidence manifest grammar does not reject extra/unexpected tokens, inconsistent case, absolute paths, or symlinked entries. | A manifest can be crafted (deliberately or accidentally) to include content the grammar was not designed to admit, weakening manifest integrity guarantees. |
| RB-022 | Validator output policy ambiguous | The validator's contract for exit codes and stdout/stderr content on pass/fail/partial outcomes is not documented or enforced consistently. | Callers of the validator (human or automated) cannot reliably distinguish outcome classes from its output alone. |
| RB-023 | Verdict parsing case-insensitive / ambiguous | The logic that parses a recorded verdict string treats case variants and near-matches as equivalent rather than requiring an exact match against an explicit allowed-token list. | A verdict field could be misread (e.g., a typo or casing variant of "READY") and be accepted as a different verdict than intended. |
| RB-024 | Manual Witness form fields under-validated | Fields on the manual Witness submission form are not validated against required formats (e.g., commit length/hex-only, hash length/hex-only, required non-empty fields) before being accepted. | Malformed or incomplete manual submissions can be accepted as if they were well-formed evidence. |
| RB-025 | Classification precedence incomplete for identity mismatches / missing artifact | The documented precedence rules for classifying a submission do not fully specify which classification wins when both an identity mismatch (wrong commit/tag/image) and a missing artifact are observed simultaneously, or similar overlapping-defect cases. | Ambiguous precedence can lead to inconsistent classification of otherwise identical defect combinations across different reviewers or runs. |
| RB-026 | Package readiness / maintainer intake / redaction / correction ledger / PR identity procedures incomplete | Several adjacent process documents — package-readiness sign-off, maintainer intake of a Witness submission, redaction rules, a correction ledger for amending prior submissions, and PR-identity verification — are not fully specified end-to-end. | Gaps in these adjacent procedures create downstream ambiguity even once the core build/evidence pipeline is otherwise sound. |
| RB-027 | Unit tests too weak; shared generic fixture bodies | The validator's unit tests reuse the same generic fixture body across multiple test cases for different required files, so tests do not actually exercise file-specific content requirements. | Passing unit tests provide weaker assurance than they appear to, because they do not distinguish between file-specific and generic content. |

All nine items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
