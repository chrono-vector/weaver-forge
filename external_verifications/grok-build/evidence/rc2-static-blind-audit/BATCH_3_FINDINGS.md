# RC2 static blind audit — Batch 3 findings: Container build & evidence outcomes

Batch 3 of 4. Static review only; no execution performed. Scope: in-container toolchain validation, evidence-file lifecycle, build-outcome modeling, error trapping, and static inspection/logging rigor for tag `grok-build-witness-v1.0.0-rc2` (`255b357c9ee33c4a9e34b5d9b6e396c53cfe494e`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RB-012 | Rustc/Cargo/DotSlash version not hard-validated | Toolchain version probes (including `DotSlash --version \|\| true`) are wired to tolerate failure rather than to hard-fail the run when the observed version does not match the pinned expectation. | A wrong or missing toolchain version can pass through unnoticed and be recorded alongside a build result that looks legitimate. |
| RB-013 | Required evidence files not initialized before fallible ops | Evidence files that the validator later requires are not created (even as placeholders) before the first fallible container operation runs. | Any early failure inside the container leaves gaps in required evidence rather than a clear, complete failure record. |
| RB-014 | No outcome-sensitive build-result model | The container build has no explicit state model distinguishing `BUILD_NOT_STARTED`, `CARGO_FAILED`, `CARGO_SUCCEEDED_ARTIFACT_MISSING`, `CARGO_SUCCEEDED_ARTIFACT_PRESENT`, and `INFRASTRUCTURE_FAILURE`; outcomes are inferred loosely from exit codes and file presence rather than recorded explicitly. | Ambiguous outcome states make it possible to misclassify a failed or incomplete run as a successful build. |
| RB-015 | Unexpected ERR traps incomplete | Shell error traps do not comprehensively cover all points at which the container script could exit unexpectedly, so some failure paths do not guarantee that outcome/evidence state is written before exit. | Certain failures can terminate the run silently, leaving no recorded explanation of what happened. |
| RB-016 | Cargo exit 0 with missing artifact can still look like success | A `cargo build` exit code of 0 is not cross-checked against actual presence of the expected artifact on disk before the run is classified as successful. | A build can be recorded as successful when the expected binary was never produced. |
| RB-017 | Required static inspection commands use `\|\| true`; `ldd`/product still prohibited | Static inspection commands intended to be mandatory are suffixed with `\|\| true`, making their failure silently non-fatal, while the prohibition on running `ldd` or the product binary itself remains correctly enforced. | Mandatory inspection steps can silently fail to run without affecting the recorded outcome, weakening the evidentiary value of "static inspection passed." |
| RB-018 | `DOCKER_EXIT_CODE.txt` bare numeric line; `BUILD_TIMING` incomplete | `DOCKER_EXIT_CODE.txt` records only a bare numeric exit code with no labeling/context, and `BUILD_TIMING` evidence does not consistently capture start time, end time, and duration together. | Evidence files lack the structure needed for unambiguous automated or manual review of timing and exit status. |

All seven items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
