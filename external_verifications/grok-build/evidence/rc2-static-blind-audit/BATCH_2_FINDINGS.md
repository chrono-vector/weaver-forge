# RC2 static blind audit — Batch 2 findings: Host orchestrator safety & acquisition

Batch 2 of 4. Static review only; no execution performed. Scope: host orchestrator input safety, package acquisition integrity, and Docker acquisition/identity enforcement for tag `grok-build-witness-v1.0.0-rc2` (`255b357c9ee33c4a9e34b5d9b6e396c53cfe494e`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RB-005 | `WITNESS_ID` unrestricted | The host orchestrator accepts `WITNESS_ID` without an allow-list; values containing a path separator, `..`, or whitespace are not rejected before the value is used to construct on-disk paths. | Opens a path-injection surface into the Witness's own filesystem layout via an otherwise-innocuous identity field. |
| RB-006 | `WORK_ROOT` deletion/reset insufficiently guarded | Destructive reset/cleanup of `WORK_ROOT` does not hard-reject dangerous values such as `/`, the user's home directory, other system prefixes, the Witness's own package/repository checkout, or a path that resolves (via symlink) into any of those. | A single misconfigured or malicious `WORK_ROOT` value could cause the orchestrator to delete far more than the intended scratch directory. |
| RB-007 | Package clone HEAD/clean status not enforced after detached checkout | After the orchestrator checks out the pinned tag/commit in detached-HEAD mode, it does not re-verify that `HEAD` matches the expected commit and that the working tree is clean before proceeding. | A silently mismatched or dirtied checkout could be built and evidenced as if it were the pinned rc2 identity. |
| RB-008 | No direct Cargo.lock SHA-256 enforcement before/after Docker | The orchestrator does not compute and compare a SHA-256 of `Cargo.lock` immediately before handing the source to Docker and again immediately after the container build, independent of any in-container checks. | Dependency-lock drift introduced by the container step (or a stale/mismatched lock file) can go undetected at the host level. |
| RB-009 | `docker pull` non-fatal; continues with cached image | If `docker pull` of the pinned image fails or is skipped, the orchestrator proceeds using whatever image is already cached locally rather than treating the failure as fatal. | The resulting build may silently run against an unpinned or stale image while evidence still claims the pinned image was used. |
| RB-010 | Image identity (RepoDigest/platform) not hard-stop enforced before `docker run` | The pinned image's RepoDigest and platform are not verified as a hard precondition immediately before `docker run`; a mismatch does not halt execution. | The pinned toolchain/image guarantee can be silently violated at the exact point where it matters most. |
| RB-011 | Evidence files missing on early failure paths | If the orchestrator fails early (e.g., during acquisition or Docker pull), the required evidence files for that stage are not guaranteed to exist, even as failure placeholders. | A failed run can leave an evidence directory that looks incomplete rather than clearly recording what failed and when, complicating post-hoc classification. |

All seven items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
