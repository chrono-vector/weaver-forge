# RC3 static blind audit — Batch 1 findings: Release identity, wording, public status

Batch 1 of 4. Static review only; no execution performed. Scope: package/tag identity readiness for a successor, normative-document wording stability, URL-mismatch classification consistency, public status currency, and detached-HEAD evidence for tag `grok-build-witness-v1.0.0-rc3` (`77221a224bbd6194cfafb81f6ecb58c800e5bc13`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RC3B-001 | Package identity still `1.0.0-rc3`; successor `rc4` identity not prepared | The package version, `WEAVER_FORGE_TAG`, and related identity constants all still resolve to `1.0.0-rc3` / `grok-build-witness-v1.0.0-rc3`. No draft `rc4` identity (version string, tag name, commit target, `RUST_IMAGE` bump if applicable) has been staged anywhere in a clearly labeled, not-yet-cut location. | Remediation work for the blockers in this audit has no distinct successor identity to be validated against; without a prepared `rc4` identity, remediation risks being conflated with the still-immutable, already-audited `rc3` identity. |
| RC3B-002 | Normative docs may retain time-unstable tag-absence / pending wording | The runbook, submission procedure, and related normative documents may still contain conditional phrasing written for a pre-publication state (e.g., "if the tag has not yet been created", "pending publication") that is no longer accurate now that `rc3` exists as a published, immutable snapshot. | Post-publication normative text written in pre-publication conditional language is ambiguous about whether the tag currently exists and is fixed, which is exactly the kind of ambiguity a blind Witness cannot safely resolve on their own. |
| RC3B-003 | Weaver Forge URL mismatch not treated as material FAIL consistently across script and classification | Detection of a `WEAVER_FORGE_URL` mismatch exists in some script paths and is described in the written classification precedence, but the two are not enforced with matching fatal weight; a run against the wrong upstream URL is not guaranteed to be classified FAIL end-to-end. | Repository-identity divergence is one of the most severe possible identity failures (wrong fork or wrong project entirely); inconsistent enforcement creates a path for such a run to receive a less severe classification than the underlying defect warrants. |
| RC3B-004 | Public status / `CLAIM_REGISTER` still describe rc3 as current candidate without recording this audit's intake | `RESULTS.md`, `CLAIM_REGISTER.md`, and related public status documents continue to describe `rc3` as the current candidate but have not been updated to record that this static blind audit occurred or that it returned **NOT READY**. | A reader who consults only the public status documents (rather than this evidence directory) would have no way to learn that this audit exists or what it found, effectively making the audit invisible outside of `external_verifications/grok-build/evidence/`. |
| RC3B-005 | Grok Build detached-HEAD check uses only symbolic-ref inconsistently; missing explicit `grok_build_detached_head=yes\|no` evidence field | After the orchestrator checks out the pinned tag/commit, detection of detached-HEAD state via `git symbolic-ref -q HEAD` is applied in some code paths only, and no evidence file records an explicit `grok_build_detached_head=yes\|no` field after the checkout completes. | Without a recorded, explicit evidence field, a post-hoc reviewer cannot machine-verify from the evidence alone that the build ran against the pinned, non-branch-tracking commit rather than a moving branch reference. |

All five items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
