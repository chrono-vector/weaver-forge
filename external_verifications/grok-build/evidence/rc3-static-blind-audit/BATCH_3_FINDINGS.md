# RC3 static blind audit — Batch 3 findings: Container evidence schemas

Batch 3 of 4. Static review only; no execution performed. Scope: key-set completeness and drift across container-produced evidence files, static-inspection-failure outcome contract, and evidence-inventory completeness gating for tag `grok-build-witness-v1.0.0-rc3` (`77221a224bbd6194cfafb81f6ecb58c800e5bc13`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RC3B-013 | `ENVIRONMENT.txt` generator/template/validator key drift; missing required keys | The keys written by the generator, declared by the template, and checked by the validator for `ENVIRONMENT.txt` have drifted relative to one another, and required keys including `status`, `outcome`, `host_kernel`, `host_cpu`, `docker_context`, `wsl2_indicator`, `ai_assistance_*`, `human_review_completed`, and `upstream_product_commands_not_run` are missing from at least one of the three. | A file this central to establishing the run's execution context cannot be reliably schema-validated while its required key set differs between the three places that are each supposed to define it. |
| RC3B-014 | `CLEAN_TARGET_PROOF.txt` uses ambiguous/duplicate `observed_entry_count`; missing distinct host/container path and UTC fields | The `observed_entry_count` field appears without being scoped to a single, unambiguous check, and the file does not separately record the host-side path checked, the container-side path checked, and a UTC timestamp for each check. | An ambiguous or duplicated count makes it impossible to tell which "clean target" claim (host pre-build vs. container pre-build) a given value refers to, undermining the file's purpose as proof of a clean starting state. |
| RC3B-015 | `BUILD_ENVIRONMENT.txt` key names drift (`HOME` vs `home`, etc.); missing `workdir`, `bootstrap_cargo_target_dir`, `grok_build_commit`, `expected_cargo_lock_sha256`, `canonical_build_command`, `mounts` | Key casing is inconsistent between generator and template/validator, and several fields needed to independently verify the exact build environment used are absent entirely. | Missing `canonical_build_command` and `expected_cargo_lock_sha256` in particular remove the ability to verify, from this file alone, that the expected command and dependency lock were actually used, independent of any other evidence file. |
| RC3B-016 | `BUILD_TIMING.txt` uses generic `utc_start`/`utc_end`; missing `docker_elapsed_seconds`, `cargo_started`, `cargo_exit_code`, `docker_exit_code`, `failure_stage`, `status` | The file records only bare start and end timestamps with no derived elapsed-duration field and no per-stage markers. | This leaves `BUILD_TIMING.txt` unable to independently corroborate the outcome recorded elsewhere (see RC3B-009); the corresponding rc2 finding (`RB-018`) remains `NOT_STARTED` heading into rc3. |
| RC3B-017 | `STATIC_ARTIFACT_INSPECTION.txt` uses repeated generic `command=`/`exit_code=` pairs; missing per-tool fields; multiline encoding undefined | The same generic key pair is reused for each inspection tool run (e.g., `file`, `nm`, `strings`) rather than namespaced per-tool keys (`file_command`/`file_output`/`file_exit_code`, etc.), and no rule defines how multiline tool output should be encoded inside a key=value evidence file. | Without per-tool fields, a validator or reviewer cannot attribute a given exit code or output to the specific inspection tool that produced it, and undefined multiline encoding risks corrupting or truncating parsed evidence. |
| RC3B-018 | Static inspection command failure does not keep `outcome=CARGO_SUCCEEDED_ARTIFACT_PRESENT` with `inspection_complete=no`, `status=FAILED`, a dedicated nonzero exit, a `PARTIAL` ceiling, and `PASS` prohibited | When a required static inspection command itself fails, the file does not enforce that the build outcome is left unchanged (i.e., not conflated with the inspection failure), that `inspection_complete` is set to `no`, that `status` is set to `FAILED`, that a dedicated nonzero exit code is recorded for the inspection stage, that the run's classification ceiling is capped at `PARTIAL`, and that `PASS` is explicitly prohibited in this case. | This is the direct successor of `RB-017` from the rc2 audit; rc3 requires a fully specified failure-state contract for this file rather than merely tolerating failure, and none of the six listed guarantees is currently enforced. |
| RC3B-019 | `POST_BUILD_INTEGRITY.txt` missing `status`, `outcome`, `source_head_unchanged`, `cargo_lock_unchanged`, `cargo_lock_post_matches_expected`, `failure_stage`; host gate does not require all four `yes` fields; blank porcelain may pollute `yes`/`no` | The file lacks the listed fields, the host-side gate that is supposed to require all four affirmative integrity fields does not actually enforce that all four are `yes` before proceeding, and an empty `git status --porcelain` result can be misinterpreted as a `yes`/`no` value rather than being explicitly normalized. | This file is the last line of defense against post-build source or dependency tampering going undetected; both the missing fields and the incomplete gating weaken that defense at the point it matters most. |
| RC3B-020 | `evidence_inventory_complete` may be marked `yes` before automated+manual+manifest are all complete; `BOOTSTRAP_PROTOC_VERSION.txt` still written under `EVIDENCE_DIR` | `evidence_inventory_complete=yes` can be set based on a subset of the three required completeness checks (automated evidence present, manual Witness-supplied fields present, manifest validates cleanly) rather than requiring all three; separately, `BOOTSTRAP_PROTOC_VERSION.txt` continues to be written directly under `EVIDENCE_DIR` rather than a designated auxiliary subpath. | A premature `evidence_inventory_complete=yes` can allow an incomplete submission to proceed to classification, and an auxiliary bootstrap file living alongside required evidence blurs the required/auxiliary boundary this program depends on (see also RC3B-021). |

All eight items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
