# RC3 static blind audit — Batch 4 findings: Validator, redaction, tests, policy

Batch 4 of 4. Static review only; no execution performed. Scope: closed-inventory auxiliary-file policy, key=value parsing strictness, manual verdict cross-checking, machine-computed verdict-ceiling consistency, redaction completeness, correction-ledger/policy alignment, and generator/schema/manifest/unit-test coverage for tag `grok-build-witness-v1.0.0-rc3` (`77221a224bbd6194cfafb81f6ecb58c800e5bc13`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RC3B-021 | Auxiliary-file policy open/ambiguous; undeclared auxiliaries not closed-inventory enumerated; undeclared files may pass if listed in manifest | Files such as `IMAGE_PULL_*`, `CARGO_LOCK_INTEGRITY`, and `BOOTSTRAP_PROTOC_VERSION.txt` are not enumerated in a closed, exhaustive allow-list of permitted auxiliary files; the validator's manifest check appears to accept any file as "auxiliary" merely because it is listed in the manifest, rather than checking it against a fixed list. | Without a closed-inventory allow-list, an evidence directory could accumulate arbitrary undeclared files that the validator waves through simply because they were added to the manifest, defeating the purpose of a fixed evidence schema. |
| RC3B-022 | `parse_kv` last-value-wins on duplicate keys; no rejection of same-key duplicates | The key=value parsing routine shared across generator/validator scripts silently takes the last occurrence of a duplicated key rather than rejecting the file as malformed. | A file with an accidental or deliberately duplicated key (e.g., two `status=` lines with different values) is silently resolved to one value with no indication a duplicate existed, hiding a potential integrity problem in the evidence file itself. |
| RC3B-023 | `WITNESS_VERDICT.md` does not require/cross-check `outcome=` against automated outcome vocabulary | The manually authored Witness verdict document accepts a free-form outcome value without cross-checking it against the fixed outcome vocabulary already recorded in the automated evidence. | A manually authored verdict can disagree with the automated record it is supposed to summarize, and nothing in the current pipeline catches that disagreement before submission. |
| RC3B-024 | Verdict ceiling not machine-computed from outcome+identity+deviation+inspection+product/ldd/independence; script ceiling may diverge from validator/`CLASSIFICATION` | The maximum permissible classification ("ceiling") for a submission is not derived by a single deterministic function of the automated outcome, identity-check results, noncanonical-deviation flag, inspection completeness, product/`ldd`-execution-prohibition compliance, and Independent-Witness status; the host/container scripts and the separate validator/`CLASSIFICATION.md` document each compute or describe a ceiling independently, with no guarantee they agree. | This is the most consequential blocker in this batch: divergent ceiling logic between script and documentation is exactly the kind of gap that could let a run be classified more favorably in one place than the underlying evidence actually supports. |
| RC3B-025 | Redaction never-redact list incomplete; marker reconciliation for `[REDACTED:` incomplete | The "never redact" list of fields that must always remain visible in any redacted evidence copy omits `outcome`, `build_status`, `failure_stage`, `proposed verdict`, `intake verdict`, `canonical_run`, and `verdict_ceiling`; no reconciliation step checks every `[REDACTED: ...]` marker inserted into a redacted copy against a corresponding logged redaction entry. | A redacted submission could hide exactly the fields a reviewer most needs to assess readiness, and unlogged redaction markers could not be distinguished from an incomplete or tampered document. |
| RC3B-026 | Correction-ledger intake states not aligned with `MAINTAINER_INTAKE_POLICY` (incl. `SUPERSEDED` include/exclude) | The correction ledger's set of intake states does not match the states defined in `MAINTAINER_INTAKE_POLICY.md`, and it is unspecified whether a `SUPERSEDED` correction-ledger entry should be included in or excluded from the active/current evidence set for classification purposes. | Mismatched vocabularies between two documents that both govern how corrections are tracked can result in a correction being processed differently depending on which document a reviewer consults. |
| RC3B-027 | No generator/schema contract tests comparing shell generator key sets to templates/`FILE_REQUIRED_FIELDS`; golden fixtures missing for all outcomes/failure modes | No automated test compares, key-by-key, the fields each shell generator actually writes against the corresponding template's/`FILE_REQUIRED_FIELDS` declaration, and there are no golden fixture files covering each outcome/failure-mode state (including those introduced to close RC3B-014, RC3B-016, RC3B-018, and RC3B-019, and their rc2 predecessors). | This is the automated safeguard that would have caught essentially every schema-drift blocker in Batch 3 (and `RB-019` from rc2) before it reached a live submission; its absence means schema drift will keep recurring across releases. |
| RC3B-028 | Manifest/unit tests incomplete (one-space separator, duplicate keys, aux policy, post-lock mismatch, verdict ceilings, unlogged redaction, `upstream_product_commands_not_run`) | Existing unit tests do not cover: the manifest's required one-space key/value separator grammar, rejection of duplicate keys (RC3B-022), enforcement of the closed auxiliary-file allow-list (RC3B-021), detection of a post-build `Cargo.lock` mismatch (RC3B-019), correct computation of the machine-derived verdict ceiling (RC3B-024), detection of an unlogged redaction marker (RC3B-025), or the presence/validation of `upstream_product_commands_not_run` (RC3B-012). | This is a comprehensive gap across the exact set of new and carried-forward defects this audit records; without test coverage for each, remediation of the underlying blockers cannot be verified to have actually closed them. |

All eight items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
