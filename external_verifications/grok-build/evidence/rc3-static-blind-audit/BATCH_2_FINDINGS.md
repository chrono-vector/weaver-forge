# RC3 static blind audit — Batch 2 findings: Host orchestrator — image, pre-Docker failure, outcomes

Batch 2 of 4. Static review only; no execution performed. Scope: image-identity evidence completeness, pre-Docker infrastructure-failure evidence, authoritative outcome determination, cross-file consistency, managed-cleanup safety, host unit-test coverage, and upstream-product-non-execution evidence for tag `grok-build-witness-v1.0.0-rc3` (`77221a224bbd6194cfafb81f6ecb58c800e5bc13`).

| ID | Finding | Detail | Why it is material |
|----|---------|--------|---------------------|
| RC3B-006 | `IMAGE_IDENTITY.txt` lacks `status=OK\|FAILED\|NOT_REACHED` and `failure_stage`; incomplete inspect command/exit fields; `status=OK` not gated on all identity checks | The file does not record a top-level `status` field with the three defined values, nor a `failure_stage` field; the underlying `docker inspect` command and its exit code are not both fully recorded; and `status=OK` can be written even when not every identity sub-check (RepoDigest, platform, architecture) has actually passed. | A partial identity check that fails on one sub-check but passes on others could still be recorded as `status=OK`, masking a genuine image-identity violation at the exact evidence file meant to attest to image pinning. |
| RC3B-007 | Pre-Docker infrastructure failures do not finalize all mandatory automated evidence with `INFRASTRUCTURE_FAILURE` before abort; may leave `NOT_REACHED` as final | When pull, inspect, digest, OS, architecture, or platform checks fail before `docker run`, the host script aborts without guaranteeing that every mandatory automated evidence file is finalized to `outcome=INFRASTRUCTURE_FAILURE`; some files may be left recording `NOT_REACHED` as their terminal value. | `NOT_REACHED` left as a final value is ambiguous between "this stage was never attempted" and "the run failed before this stage could become authoritative," weakening the completeness and authoritativeness the evidence set is supposed to guarantee on every exit path. |
| RC3B-008 | Host may reconstruct ordinary outcomes from `cargo_started`/artifact/docker exit alone; `BUILD_EXIT_CODE.txt` container outcome not strictly authoritative after Docker begins | Once the container has started, the host script infers an outcome by combining `cargo_started` markers, artifact presence, and the raw Docker exit code, rather than treating the outcome value written inside the container (`BUILD_EXIT_CODE.txt`) as the single source of truth once it exists. | Independent host-side re-derivation of the outcome can diverge from the container's own recorded outcome, producing two different "truths" about the same run and undermining which evidence file a reviewer should trust. |
| RC3B-009 | Outcome consistency not machine-enforced across `BUILD_EXIT_CODE` / `BUILD_TIMING` / `DOCKER_EXIT_CODE` / `ARTIFACT_IDENTITY` / `STATIC_ARTIFACT_INSPECTION` / `WITNESS_VERDICT` | No automated step cross-checks the outcome/status fields recorded across these six evidence files against one another before a submission is treated as complete. | Contradictory evidence between files (for example, a build-success outcome alongside `inspection_complete=no`) can persist undetected all the way into a final submission and classification. |
| RC3B-010 | Managed child `rm -rf` may follow symlinks; no lstat/readlink-before-delete policy | The host orchestrator's cleanup of managed child/scratch directories under `WORK_ROOT` uses `rm -rf` without first `lstat`-ing the target or rejecting symlinked entries, so a symlink planted inside the managed tree could redirect deletion outside the intended scratch area. | This is a destructive-operation safety gap in the managed-child cleanup path specifically (distinct from, and not fully closed by, the `WORK_ROOT`-level guard remediation carried forward from the rc2 audit). |
| RC3B-011 | Host safety unit tests incomplete for constants, overrides, `WORK_ROOT`, dirty clones, lock/image failures, outcome preservation | No unit test suite exercises the frozen canonical identity constants and override-disclosure behavior, `WORK_ROOT` guard rejections, detached-HEAD dirty-clone detection, `Cargo.lock` mismatch handling, image pull/identity failure handling, or outcome-preservation logic in isolation from an actual Docker run. | Safety-critical host logic remains unverified by any repeatable, execution-free test harness, so regressions in any of these paths would go undetected without running the full pipeline — which this audit program is specifically designed to avoid requiring. |
| RC3B-012 | `upstream_product_commands_not_run` not required/validated in `ENVIRONMENT` / `WITNESS_STATEMENT` / `WITNESS_VERDICT` | None of `ENVIRONMENT.txt`, the Witness statement, or `WITNESS_VERDICT.md` require or validate an explicit `upstream_product_commands_not_run=yes\|no` field attesting that the `xai-grok-pager` product itself, or `ldd` against its artifact, was not executed by the Witness. | Without this explicit, validated field, a submission provides no positive, checkable evidence of adherence to the "no product execution" constraint that the entire static-audit program depends on. |

All seven items in this batch are treated as material blockers to a READY verdict. See `INTEGRATED_BLOCKERS.md` for consolidated IDs and `INTEGRATED_REMEDIATION_LIST.md` for mapped remediation steps.
